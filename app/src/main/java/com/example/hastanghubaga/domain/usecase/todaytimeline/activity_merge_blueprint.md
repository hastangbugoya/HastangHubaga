## Core Concepts

### 1. Template
User-defined reusable definition.

Example:
- "Run"
- "Sleep"

Stored as:
- ActivityEntity

---

### 2. Occurrence (Planned)

A **materialized instance for a specific date**.

Generated by:
- `MaterializeActivityOccurrencesForDateUseCase`

Stored as:
- `ActivityOccurrenceEntity`

Key properties:
- `occurrenceId` (stable ID per planned instance)
- `scheduledTime`

---

### 3. Logged Event (Actual)

What the user actually did.

Created via:
- `SaveExerciseActivityUseCase`

Stored as:
- ActivityLogEntity

Key properties:
- `occurrenceId` (nullable but SHOULD be set when coming from planned)
- `startTimestamp`
- `endTimestamp`

---

## Critical Invariant

> **One occurrence â†’ at most one log**

This is enforced via:
- upsert by `occurrenceId`

---

## Merge Strategy (Timeline)

The timeline is built using:

```
BuildTodayTimelineUseCase
```

### Inputs:
- Planned occurrences
- Logged activities

---

## Merge Rules

### Case 1: Planned only (not logged)

Show:
```
ActivityTimelineItem(
    isCompleted = false
)
```

---

### Case 2: Logged with occurrenceId

**Replace planned occurrence**

Show ONLY:
```
ActivityTimelineItem(
    isCompleted = true
)
```

Do NOT show planned item.

---

### Case 3: Logged without occurrenceId (force log)

Show:
```
ActivityTimelineItem(
    isCompleted = true
)
```

AND keep planned items untouched.

---

## Matching Logic

```
log.occurrenceId == occurrence.occurrenceId
```

If match:
â†’ planned is consumed
â†’ replaced by log

---

## Timeline Assembly Flow

1. Load occurrences
2. Load logs
3. Index logs by occurrenceId
4. Iterate occurrences:
   - if match â†’ use log
   - else â†’ use planned
5. Add orphan logs (no occurrenceId)
6. Sort by time

## Key Design Principles

1. Never mutate planned data when logging
2. Logs represent truth
3. Occurrence is the bridge
4. Timeline is a projection layer
