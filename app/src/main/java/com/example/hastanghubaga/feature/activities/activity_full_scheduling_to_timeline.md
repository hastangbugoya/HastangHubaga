# Activity Scheduling → Timeline Merge (Full System Blueprint)

**Document Version Date:** 2026-04-04

---

# Purpose

This document describes the **end-to-end activity system**:

**Scheduling → Occurrence → Logging → Timeline Merge**

This is the **authoritative blueprint** for:
- how activities flow through the system
- how planned and actual data reconcile
- how the timeline is built

This is the **reference implementation** for all future systems (meals, supplements).

---

# High-Level Flow

```
Activity Template
    ↓
Scheduling Rules
    ↓
Occurrence Materialization
    ↓
User Logging (Actual)
    ↓
Timeline Merge
```

---

# 1. Activity Template Layer

### Table: `activity`

Defines reusable activity definitions.

Key fields:
- `id`
- `type`
- `notes`
- `isActive`
- `isWorkout`

### Rule
- Only `isActive = true` templates generate planned occurrences
- Logs remain valid even if template becomes inactive later

---

# 2. Scheduling Layer

### Tables:
- `activity_schedules`
- `activity_schedule_fixed_times`
- `activity_schedule_anchored_times`

### Responsibilities
Scheduling determines:

- does this activity occur on this date?
- what time should it occur?

### Supported Scheduling Types

| Type | Description |
|------|------------|
| Fixed Time | Exact time (e.g. 7:00 AM) |
| Weekly | Specific weekdays |
| Interval | Every X days |
| Anchored | Relative to events (future-ready) |

---

# 3. Occurrence Layer

### Table: `activity_occurrences`

Materialized planned instances.

Key fields:
- `id` (**occurrenceId**)
- `activityId`
- `date`
- `plannedTimeSeconds`
- `isWorkout`

### Generated by:
`MaterializeActivityOccurrencesForDateUseCase`

### Key Principle
> Occurrence = planned intent, NOT execution

---

# 4. Logging Layer (Actual)

### Table: `activity_logs`

Represents what actually happened.

Key fields:
- `id`
- `activityId`
- `occurrenceId` (nullable)
- `startTimestamp`
- `endTimestamp`
- `notes`
- `intensity`

---

## Linked vs Unlinked Logs

| Type | Meaning |
|------|--------|
| With occurrenceId | Fulfills a planned occurrence |
| Without occurrenceId | Manual / extra / ad-hoc |

---

# 5. Timeline Merge Layer

### Use Case:
`BuildTodayTimelineUseCase`

---

## Step-by-Step Merge

### Step 1 — Build Planned Items

From `activity_occurrences`:

```
isCompleted = false
occurrenceId = occurrence.id
time = plannedTime
```

---

### Step 2 — Build Actual Items

From `activity_logs`:

```
isCompleted = true
time = log.start
occurrenceId = log.occurrenceId OR syntheticId
```

---

### Step 3 — Compute Satisfaction

```
satisfiedIds = logs.mapNotNull { it.occurrenceId }
```

---

### Step 4 — Suppress Planned

```
planned.filterNot { it.occurrenceId in satisfiedIds }
```

---

### Step 5 — Merge

```
final = remainingPlanned + actual
```

---

### Step 6 — Sort

Sorted by:
1. time
2. type
3. stable keys

---

# Merge Behavior

| Scenario | Result |
|--------|-------|
| Planned only | Show planned |
| Planned + logged (linked) | Replace with logged |
| Logged only (no occurrenceId) | Show logged |
| Template inactive | Stop planning, keep logs |

---

# OccurrenceId — The Core Concept

OccurrenceId is:

> The unique identity of ONE planned instance on ONE day

### Why it matters:
- enables exact replacement
- prevents duplicate rows
- avoids time-based bugs

---

# Upsert Rule

For logs with occurrenceId:

> One occurrence → one log

Use upsert behavior.

---

# UI Semantics

| Row Type | isCompleted | Meaning |
|----------|------------|--------|
| Planned | false | Pending |
| Logged | true | Completed |

---

# Critical Design Rules

1. **Do NOT match by time**
2. **Do NOT match by title**
3. **Only match by occurrenceId**
4. **Do NOT delete planned rows — suppress them**
5. **Do NOT delete logs — they are truth**
6. **Manual logs must always appear**
